# Activity 1.7.1 - Deny User by Default Policy

## ZIG Requirement Summary

Per NSA ZIG Phase One, Activity 1.7.1 requires:
- Applications updated to deny-by-default
- Reduced default permission levels (least privilege)
- Audit and revoke unnecessary access
- Use roles/attributes for access instead of static permissions
- Automated audit and governance functions

## Checks Implemented

### Check 1.7.1-A: Overly Permissive IAM Policies (Action:* Resource:*)

| Field | Value |
|-------|-------|
| Severity | HIGH |
| AWS API | `iam list-policies`, `iam get-policy-version` |
| Fields Checked | Policy document Action and Resource fields |

**What it checks:** Identifies customer-managed IAM policies that grant `Action: "*"` with `Resource: "*"` (full admin equivalent).

**Why it matters:** ZIG requires "reduced default permission levels" and least privilege. Policies with `*:*` permissions:
- Violate least privilege principle
- Grant far more access than typically needed
- Increase blast radius if credentials are compromised

**Pass condition:** No customer-managed policies with `Action: "*"` and `Resource: "*"`

**Fail condition:** One or more policies grant unrestricted access

**Note:** AWS-managed policies (like AdministratorAccess) are not flagged here - they're checked in 1.4.1-A for user attachment.

**Remediation:**
```bash
# List customer policies
aws iam list-policies --scope Local --query 'Policies[].Arn'

# Get policy document
aws iam get-policy-version --policy-arn <arn> --version-id v1

# Replace with scoped permissions
# Example: Instead of "Action": "*", use specific actions like:
# "Action": ["s3:GetObject", "s3:PutObject"]
```

### Check 1.7.1-B: IAM Policies with Service Wildcards on All Resources

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `iam list-policies`, `iam get-policy-version` |
| Fields Checked | Policy document for patterns like `"s3:*"` with `Resource: "*"` |

**What it checks:** Identifies policies that grant all actions for a service (e.g., `s3:*`, `ec2:*`) on all resources.

**Why it matters:** While not as severe as `*:*`, service-wide wildcards still violate least privilege:
- `s3:*` grants delete, not just read
- `ec2:*` grants terminate, not just describe
- Users typically need a subset of service actions

**Pass condition:** No policies with `service:*` on `Resource: "*"`

**Fail condition:** Policies found with service-wide wildcards

**Remediation:**
```bash
# Scope to specific actions needed
# Instead of: "Action": "s3:*"
# Use: "Action": ["s3:GetObject", "s3:ListBucket"]

# Scope to specific resources
# Instead of: "Resource": "*"
# Use: "Resource": "arn:aws:s3:::my-bucket/*"
```

### Check 1.7.1-C: IAM Access Analyzer Enabled

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `accessanalyzer list-analyzers` |
| Fields Checked | Active analyzer count |

**What it checks:** Whether IAM Access Analyzer is enabled for automated permission auditing.

**Why it matters:** ZIG requires "audit functions and governance processes are implemented and automated." Access Analyzer provides:
- Automated detection of external access to resources
- Unused access analysis (identifies permissions not being used)
- Policy validation
- Policy generation based on actual usage

**Pass condition:** At least one active IAM Access Analyzer

**Fail condition:** No Access Analyzer configured

**Remediation:**
```bash
# Create an account-level analyzer
aws accessanalyzer create-analyzer \
  --analyzer-name account-analyzer \
  --type ACCOUNT

# Or organization-level analyzer (from management account)
aws accessanalyzer create-analyzer \
  --analyzer-name org-analyzer \
  --type ORGANIZATION
```

### Check 1.7.1-D: Users with Direct Policy Attachments

| Field | Value |
|-------|-------|
| Severity | LOW |
| AWS API | `iam list-users`, `iam list-attached-user-policies`, `iam list-user-policies` |
| Fields Checked | Directly attached managed and inline policies |

**What it checks:** Identifies IAM users with policies attached directly (not via groups).

**Why it matters:** ZIG emphasizes RBAC - "roles/attributes for access." Direct policy attachment:
- Harder to audit and manage at scale
- Doesn't follow RBAC pattern
- Makes permission changes per-user instead of per-role
- Complicates ILM when users change roles

Best practice: Attach policies to groups, add users to groups.

**Pass condition:** No users with directly attached policies

**Fail condition:** Users have policies attached directly

**Note:** LOW severity because some direct attachments may be intentional (break-glass, specific exceptions).

**Remediation:**
```bash
# List user's direct policies
aws iam list-attached-user-policies --user-name <user>
aws iam list-user-policies --user-name <user>

# Create group with same permissions
aws iam create-group --group-name <role-name>
aws iam attach-group-policy --group-name <role-name> --policy-arn <policy>

# Add user to group, remove direct attachment
aws iam add-user-to-group --user-name <user> --group-name <role-name>
aws iam detach-user-policy --user-name <user> --policy-arn <policy>
```

### Check 1.7.1-E: S3 Account-Level Block Public Access

| Field | Value |
|-------|-------|
| Severity | HIGH |
| AWS API | `s3control get-public-access-block` |
| Fields Checked | All four block public access settings |

**What it checks:** Whether S3 Block Public Access is enabled at the account level.

**Why it matters:** ZIG requires "deny-by-default." S3 Block Public Access enforces this for S3:
- Blocks public ACLs
- Blocks public bucket policies
- Prevents accidental public exposure
- Account-level setting cannot be overridden by bucket settings

**Pass condition:** All four settings enabled: `BlockPublicAcls`, `IgnorePublicAcls`, `BlockPublicPolicy`, `RestrictPublicBuckets`

**Fail condition:** Any setting is disabled

**Remediation:**
```bash
aws s3control put-public-access-block \
  --account-id <account-id> \
  --public-access-block-configuration \
    BlockPublicAcls=true,\
    IgnorePublicAcls=true,\
    BlockPublicPolicy=true,\
    RestrictPublicBuckets=true
```

### Check 1.7.1-F: Access Analyzer Findings (External Access)

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `accessanalyzer list-findings` |
| Fields Checked | Active findings count |

**What it checks:** Whether there are active Access Analyzer findings indicating external access to resources.

**Why it matters:** Access Analyzer findings indicate resources accessible from outside the account/organization - potential violations of deny-by-default. Active findings should be reviewed and resolved.

**Pass condition:** No active Access Analyzer findings, or Access Analyzer not enabled (separate check)

**Fail condition:** Active findings exist that haven't been resolved

**Remediation:**
```bash
# List active findings
aws accessanalyzer list-findings \
  --analyzer-arn <analyzer-arn> \
  --filter '{"status": {"eq": ["ACTIVE"]}}'

# Review each finding and either:
# 1. Fix the resource policy to remove external access
# 2. Archive the finding if access is intentional
aws accessanalyzer archive-findings \
  --analyzer-arn <analyzer-arn> \
  --ids <finding-id>
```

## Checks NOT Implemented

| Requirement | Reason |
|-------------|--------|
| Application-level deny-by-default | Application code/config outside AWS API scope |
| Data classification for access control | Requires Macie or external DLP |
| Complete permission audit | Would require analyzing all policies against actual usage (complex) |
| Dynamic/rule-based access | Requires external policy engine |
