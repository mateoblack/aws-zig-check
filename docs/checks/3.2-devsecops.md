# Activity 3.2.1/3.2.2 - DevSecOps Software Factory

## ZIG Requirement Summary

Per NSA ZIG Phase One, Activities 3.2.1 and 3.2.2 require:
- Establish DevSecOps best practices for CI/CD pipelines
- Integrate vulnerability management into CI/CD pipelines
- Implement Component CI/CD pipeline(s) and Software Factory
- Adopt continual validation process/technology
- Enforce secure code signing and repository access controls

## Predecessor/Successor Relationships

```
None → 3.2.1 (DevSecOps Part 1) → 3.2.2 (DevSecOps Part 2) → 3.5.1 (cATO Part 1)
```

**Predecessor (3.2.1):** None - foundational activity
**Successor (3.2.1):** Activity 3.2.2 - DevSecOps Part 2, Activity 3.2.3 - Automate App Security
**Predecessor (3.2.2):** Activity 3.2.1 - DevSecOps Part 1
**Successor (3.2.2):** Activity 3.5.1 - Continuous ATO Part 1

## AWS Limitations

AWS provides CI/CD building blocks but not a complete DevSecOps platform. Full compliance requires:
- SAST/DAST tools integration (SonarQube, Checkmarx, etc.)
- Security testing automation beyond basic scanning
- Comprehensive code review workflows
- DevSecOps culture and process adoption

## Checks Implemented

---

### Check 3.2-A: CodePipeline Exists

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `codepipeline list-pipelines` |
| Requirement | "Implement CI/CD pipelines" |

**What it checks:** Verifies at least one CodePipeline exists for automated software delivery.

**Why it matters:** CI/CD pipelines are foundational to DevSecOps, enabling automated builds, tests, and deployments with security gates.

**Pass condition:** At least one CodePipeline pipeline exists

**Fail condition:** No CodePipeline pipelines found

**Remediation:** Create CodePipeline for automated software delivery

---

### Check 3.2-B: CodeBuild Projects

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `codebuild list-projects`, `codebuild batch-get-projects` |
| Requirement | "Integrate security testing into CI/CD" |

**What it checks:** Verifies CodeBuild projects exist for automated builds.

**Why it matters:** CodeBuild enables automated builds with security scanning integration. Projects with privileged mode can build Docker images.

**Pass condition:** At least one CodeBuild project exists

**Fail condition:** No CodeBuild projects found

**Remediation:** Create CodeBuild projects for automated builds with security scanning

---

### Check 3.2-C: ECR Image Scanning

| Field | Value |
|-------|-------|
| Severity | HIGH |
| AWS API | `ecr describe-repositories` |
| Requirement | "Vulnerability management integrated into CI/CD" |

**What it checks:** Verifies all ECR repositories have scan-on-push enabled.

**Why it matters:** Container image scanning identifies vulnerabilities in base images and dependencies before deployment.

**Pass condition:** All ECR repositories have `scanOnPush=true`

**Fail condition:** One or more ECR repositories without scan-on-push

**Remediation:** `aws ecr put-image-scanning-configuration --repository-name REPO --image-scanning-configuration scanOnPush=true`

---

### Check 3.2-D: CodeBuild Report Groups

| Field | Value |
|-------|-------|
| Severity | LOW |
| AWS API | `codebuild list-report-groups` |
| Requirement | "Track security test results" |

**What it checks:** Verifies CodeBuild report groups exist for test result tracking.

**Why it matters:** Report groups enable tracking of SAST, DAST, and unit test results over time.

**Pass condition:** At least one report group configured

**Fail condition:** No report groups configured (when CodeBuild projects exist)

**Remediation:** Configure CodeBuild report groups for test result tracking

---

### Check 3.2-E: CodeArtifact for Dependencies

| Field | Value |
|-------|-------|
| Severity | LOW |
| AWS API | `codeartifact list-domains`, `codeartifact list-repositories` |
| Requirement | "Approved repository usage" |

**What it checks:** Verifies CodeArtifact is configured for dependency management.

**Why it matters:** CodeArtifact provides a secure, approved source for software dependencies, reducing supply chain risk.

**Pass condition:** At least one CodeArtifact domain exists

**Fail condition:** No CodeArtifact configuration

**Remediation:** Configure CodeArtifact for approved package management

## Checks NOT Implemented

| Requirement | Reason |
|-------------|--------|
| SAST tool integration | Requires external tools (SonarQube, Checkmarx) |
| DAST tool integration | Requires external tools (OWASP ZAP, Burp) |
| Code review workflows | Process/culture - not API queryable |
| Secure code signing | Requires external PKI/signing infrastructure |
| DevSecOps culture adoption | Organizational - not API queryable |

## External Tools for Full Compliance

To fully implement Activities 3.2.1/3.2.2, integrate AWS with:
- **SAST** - SonarQube, Checkmarx, Fortify for static analysis
- **DAST** - OWASP ZAP, Burp Suite for dynamic testing
- **SCA** - Snyk, WhiteSource for dependency scanning
- **Code Signing** - AWS Signer, external PKI for artifact signing
- **Secret Scanning** - git-secrets, TruffleHog for credential detection

## How AWS Complements External DevSecOps

| AWS Service | Complements |
|-------------|-------------|
| CodePipeline | Orchestrates CI/CD with external tool integration |
| CodeBuild | Runs SAST/DAST tools in build environment |
| ECR | Container registry with basic vulnerability scanning |
| CodeArtifact | Approved dependency repository |
| Inspector | Continuous vulnerability scanning post-deployment |
