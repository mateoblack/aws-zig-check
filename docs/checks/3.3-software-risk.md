# Activity 3.3.1/3.3.2 - Software Risk Management

## ZIG Requirement Summary

Per NSA ZIG Phase One, Activities 3.3.1 and 3.3.2 require:
- Establish vulnerability management governance team
- Track and manage public vulnerabilities based on applications/services
- Evaluate supplier sourcing risk for approved sources
- Create Software Bill of Materials (SBOM) for applications
- Use approved vulnerability databases in DevSecOps

## Predecessor/Successor Relationships

```
None → 3.3.2 (Vuln Mgmt Part 1) → 3.3.1 (Approved Binaries) → None
                                → 3.3.3 (Vuln Mgmt Part 2)
```

**Predecessor (3.3.2):** None - foundational activity
**Successor (3.3.2):** Activity 3.3.1 - Approved Binaries, Activity 3.3.3 - Vuln Mgmt Part 2
**Predecessor (3.3.1):** Activity 3.3.2 - Vulnerability Management Part 1
**Successor (3.3.1):** None

## AWS Limitations

AWS provides vulnerability scanning but not complete vulnerability management:
- No built-in vulnerability governance workflows
- Limited SBOM generation (Inspector can export)
- No supplier risk assessment tools
- Requires external tools for complete program

## Checks Implemented

---

### Check 3.3-A: Inspector Enabled

| Field | Value |
|-------|-------|
| Severity | HIGH |
| AWS API | `inspector2 batch-get-account-status` |
| Requirement | "Vulnerability management program" |

**What it checks:** Verifies Amazon Inspector is enabled for vulnerability scanning.

**Why it matters:** Inspector provides continuous vulnerability scanning for EC2, ECR containers, and Lambda functions - foundational for vulnerability management.

**Pass condition:** Inspector status is ENABLED

**Fail condition:** Inspector not enabled

**Remediation:** `aws inspector2 enable --resource-types EC2 ECR LAMBDA`

---

### Check 3.3-B: Container Image Vulnerabilities

| Field | Value |
|-------|-------|
| Severity | BLOCKER/HIGH |
| AWS API | `inspector2 list-findings` |
| Requirement | "Track and manage vulnerabilities" |

**What it checks:** Queries Inspector for CRITICAL and HIGH severity container image findings.

**Why it matters:** Container vulnerabilities can be exploited at runtime. Critical findings require immediate remediation.

**Pass condition:** No CRITICAL or HIGH container image findings

**Fail condition:** CRITICAL findings = BLOCKER, HIGH findings = HIGH severity

**Remediation:** Update base images and dependencies, rebuild containers

---

### Check 3.3-C: ECR Lifecycle Policies

| Field | Value |
|-------|-------|
| Severity | MEDIUM/LOW |
| AWS API | `ecr get-lifecycle-policy` |
| Requirement | "Approved binaries management" |

**What it checks:** Verifies ECR repositories have lifecycle policies for image retention.

**Why it matters:** Lifecycle policies ensure old/vulnerable images are removed, maintaining only approved binaries.

**Pass condition:** All ECR repositories have lifecycle policies

**Fail condition:** ECR repositories without lifecycle policies

**Remediation:** Configure ECR lifecycle policies for image retention

---

### Check 3.3-D: Lambda Function Vulnerabilities

| Field | Value |
|-------|-------|
| Severity | BLOCKER/HIGH |
| AWS API | `inspector2 list-findings` |
| Requirement | "Track vulnerabilities in applications" |

**What it checks:** Queries Inspector for CRITICAL and HIGH severity Lambda function findings.

**Why it matters:** Lambda functions with vulnerable dependencies can be exploited. Serverless doesn't mean secure.

**Pass condition:** No CRITICAL or HIGH Lambda findings

**Fail condition:** CRITICAL findings = BLOCKER, HIGH findings = HIGH severity

**Remediation:** Update Lambda function dependencies and redeploy

---

### Check 3.3-E: SBOM Generation Capability

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `inspector2 list-coverage` |
| Requirement | "Create SBOM for applications" |

**What it checks:** Verifies Inspector is enabled and can generate SBOMs.

**Why it matters:** SBOMs identify software components, enabling supply chain risk assessment and vulnerability tracking.

**Pass condition:** Inspector enabled with coverage data available

**Fail condition:** Inspector not enabled - SBOM generation unavailable

**Remediation:** Enable Inspector, use `aws inspector2 create-sbom-export`

## Checks NOT Implemented

| Requirement | Reason |
|-------------|--------|
| Vulnerability governance team | Organizational - not API queryable |
| Supplier risk assessment | Requires external tools/processes |
| Vulnerability remediation workflows | Process - not API queryable |
| CVE database integration | Inspector uses NVD internally |
| Third-party library vetting | Requires SCA tools |

## External Tools for Full Compliance

To fully implement Activities 3.3.1/3.3.2, integrate AWS with:
- **Vulnerability Management** - Tenable, Qualys, Rapid7 for comprehensive scanning
- **SCA** - Snyk, WhiteSource, Black Duck for dependency analysis
- **SBOM Tools** - Syft, CycloneDX for SBOM generation
- **GRC Platform** - For vulnerability governance workflows
- **Threat Intelligence** - For CVE prioritization and context

## How AWS Complements External Vulnerability Management

| AWS Service | Complements |
|-------------|-------------|
| Inspector | Continuous scanning → feeds external vuln mgmt |
| ECR Scanning | Container vulnerabilities → SBOM enrichment |
| Security Hub | Aggregates findings → GRC integration |
| EventBridge | Triggers remediation workflows |
| Systems Manager | Patch deployment for remediation |
