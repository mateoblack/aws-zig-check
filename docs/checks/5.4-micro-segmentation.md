# Activity 5.4.1 - Implement Micro-Segmentation

## ZIG Requirement Summary

Per NSA ZIG Phase One, Activity 5.4.1 requires:
- Implement micro-segmentation infrastructure into SDN environment
- Enable basic segmentation of service components (web, app, DB), ports, and protocols
- Accept automated policy changes including API decision-making
- Implement distributed NGFW/micro-FW/endpoint agent in virtual hosting environments

## Predecessor/Successor Relationships

```
5.3.1 (Datacenter Macro-Segmentation) → 5.4.1 (Implement Micro-Segmentation) → 5.4.2
```

**Predecessor:** Activity 5.3.1 - Datacenter Macro-Segmentation
**Successor:** Activity 5.4.2 - Application and Device Micro-Segmentation

## AWS Limitations

AWS provides micro-segmentation through security groups and network controls:
- Security Groups act as stateful micro-firewalls per resource
- NACLs provide subnet-level stateless filtering
- EKS/ECS support network policies for container segmentation
- RDS, OpenSearch can be deployed in private subnets

Full ZIG compliance requires additional tooling for:
- Host-level process micro-segmentation
- Automated policy changes via API
- Dynamic policy engines with ML

## RAM-Shared Resources

Resources shared via AWS Resource Access Manager (RAM) are checked and reported with owner account noted. You may need to contact the resource owner to remediate issues on shared resources.

## Checks Implemented

---

### Check 5.4-A: Security Groups - Unrestricted SSH/RDP

| Field | Value |
|-------|-------|
| Severity | HIGH |
| AWS API | `ec2 describe-security-groups` |
| Requirement | "Implement distributed NGFW/micro-FW" |

**What it checks:** Identifies security groups allowing SSH (22) or RDP (3389) from 0.0.0.0/0.

**Why it matters:** Unrestricted management access violates micro-segmentation principles. Access should be limited to specific IPs or use bastion/SSM.

**Pass condition:** No security groups with 0.0.0.0/0 on SSH/RDP

**Fail condition:** Security groups allow world-open SSH/RDP

**Remediation:** 
- Restrict SSH/RDP to specific IP ranges
- Use AWS Systems Manager Session Manager for shell access
- Implement bastion hosts with restricted access

**Note:** RAM-shared security groups show owner account ID

---

### Check 5.4-B: Security Groups - Unrestricted Egress

| Field | Value |
|-------|-------|
| Severity | LOW |
| AWS API | `ec2 describe-security-groups` |
| Requirement | "Segmentation of service components" |

**What it checks:** Counts security groups with unrestricted egress (0.0.0.0/0 on all ports).

**Why it matters:** Micro-segmentation includes controlling outbound traffic. Unrestricted egress allows potential data exfiltration.

**Pass condition:** Security groups have restricted egress

**Fail condition:** Security groups allow all outbound traffic

**Remediation:** Restrict egress to required destinations and ports only

**Note:** This is common and LOW severity - many environments allow unrestricted egress

---

### Check 5.4-C: RDS in Private Subnets

| Field | Value |
|-------|-------|
| Severity | HIGH |
| AWS API | `rds describe-db-instances` |
| Requirement | "Distribute DAAS across systems with explicit access controls" |

**What it checks:** Verifies RDS instances are not publicly accessible.

**Why it matters:** Databases should be in private subnets with access only from application tier. Public databases violate micro-segmentation.

**Pass condition:** All RDS instances have PubliclyAccessible=false

**Fail condition:** RDS instances are publicly accessible

**Remediation:** 
- Modify RDS instance to disable public accessibility
- Deploy in private subnet
- Access via application tier or bastion

---

### Check 5.4-D: EKS Cluster Endpoint Access

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `eks describe-cluster` |
| Requirement | "Virtual hosting environments employ micro-segmentation at host/container level" |

**What it checks:** Verifies EKS clusters have private endpoint access enabled (not public-only).

**Why it matters:** EKS API server should be accessible from within VPC. Public-only access exposes the control plane.

**Pass condition:** EKS clusters have private endpoint access enabled

**Fail condition:** EKS clusters have public-only endpoint access

**Remediation:** Enable private endpoint access for EKS cluster

---

### Check 5.4-E: OpenSearch in VPC

| Field | Value |
|-------|-------|
| Severity | HIGH |
| AWS API | `opensearch describe-domain` |
| Requirement | "Distribute DAAS across systems with explicit access controls" |

**What it checks:** Verifies OpenSearch domains are deployed within a VPC.

**Why it matters:** OpenSearch domains outside VPC are publicly accessible. VPC deployment provides network isolation.

**Pass condition:** All OpenSearch domains are in VPC

**Fail condition:** OpenSearch domains not in VPC

**Remediation:** Create new OpenSearch domain in VPC (cannot migrate existing public domain)

## Checks NOT Implemented

| Requirement | Reason |
|-------------|--------|
| Host-level process micro-segmentation | Requires OS-level agents |
| Automated policy changes via API | Requires custom automation |
| Dynamic policy engines | Requires external tooling |
| Container network policies | Requires EKS/ECS network policy configuration |

## External Tools for Full Compliance

To fully implement Activity 5.4.1, integrate AWS with:
- **Container Security** - Calico, Cilium for Kubernetes network policies
- **Host Agents** - CrowdStrike, Carbon Black for process-level segmentation
- **Policy Automation** - Terraform, CloudFormation for infrastructure as code
- **SOAR** - AWS Security Hub, custom Lambda for automated response

## How AWS Provides Micro-Segmentation

| ZIG Concept | AWS Equivalent |
|-------------|----------------|
| Micro-FW/Endpoint Agent | Security Groups (per-resource) |
| Service Component Segmentation | Private subnets, security groups |
| Container Segmentation | EKS network policies, ECS task networking |
| API Decision Points | IAM policies, resource policies |
| Automated Policy Changes | CloudFormation, Terraform, AWS Config rules |

## Security Group Best Practices

For ZIG-compliant micro-segmentation:

1. **Principle of Least Privilege**: Only allow required ports/protocols
2. **Source Restrictions**: Use security group references instead of CIDR where possible
3. **Egress Control**: Restrict outbound to known destinations
4. **Naming Convention**: Use descriptive names indicating purpose
5. **Regular Review**: Audit unused rules and overly permissive access
