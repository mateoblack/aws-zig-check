# Activity 5.3.1 - Datacenter Macro-Segmentation

## ZIG Requirement Summary

Per NSA ZIG Phase One, Activity 5.3.1 requires:
- Implement service-based architectures to restrict lateral movement between public and private components
- Integrate proxy/enforcement checks with SDN based on device attributes and behavior
- Establish proxy/enforcement checks of attributes (device, location, data), access and flow (client, tenant, traffic patterns), and Component principles (asset life cycle, compliance, policy)

## AWS Limitations

AWS provides macro-segmentation through VPCs, subnets, and network boundaries:
- VPCs provide network isolation
- Public/private subnets separate internet-facing from internal resources
- NAT Gateways control egress from private subnets
- VPC Flow Logs provide traffic visibility
- AWS Network Firewall provides stateful inspection

Full ZIG compliance requires additional tooling for:
- SDN controller integration (AWS manages internally)
- Proxy/enforcement checks based on device attributes
- Real-time behavior-based access decisions

## RAM-Shared Resources

Resources shared via AWS Resource Access Manager (RAM) are checked and reported with owner account noted. You may need to contact the resource owner to remediate issues on shared resources.

## Checks Implemented

---

### Check 5.3-A: Public/Private Subnet Separation

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `ec2 describe-subnets` |
| Requirement | "Restrict lateral movement between public and private components" |

**What it checks:** Verifies VPCs have both public and private subnets (based on MapPublicIpOnLaunch setting).

**Why it matters:** Macro-segmentation requires separation of public-facing resources from internal resources. This is the foundation of network security architecture.

**Pass condition:** VPCs have both public and private subnets

**Fail condition:** VPCs have only public or only private subnets

**Remediation:** Design VPC architecture with public subnets (for load balancers, NAT) and private subnets (for applications, databases)

---

### Check 5.3-B: NAT Gateway for Private Subnet Egress

| Field | Value |
|-------|-------|
| Severity | LOW |
| AWS API | `ec2 describe-nat-gateways` |
| Requirement | "Control egress from private segments" |

**What it checks:** Verifies NAT Gateways exist for private subnet internet access.

**Why it matters:** NAT Gateways provide controlled egress for private resources, preventing direct internet exposure while allowing outbound connectivity.

**Pass condition:** NAT Gateway exists (or no private subnets)

**Fail condition:** Private subnets exist without NAT Gateway

**Remediation:** Deploy NAT Gateway in public subnet and route private subnet traffic through it

---

### Check 5.3-C: VPC Flow Logs Enabled

| Field | Value |
|-------|-------|
| Severity | HIGH |
| AWS API | `ec2 describe-flow-logs` |
| Requirement | "Ensure log activities are captured in SIEM" |

**What it checks:** Verifies all VPCs have flow logs enabled for traffic visibility.

**Why it matters:** Flow logs are essential for monitoring traffic patterns, detecting anomalies, and supporting incident response. ZIG requires centralized logging.

**Pass condition:** All VPCs have flow logs enabled

**Fail condition:** VPCs without flow logs

**Remediation:** Enable VPC Flow Logs to CloudWatch Logs or S3, integrate with SIEM

**Note:** RAM-shared VPCs show owner account ID - contact owner for remediation

---

### Check 5.3-D: AWS Network Firewall

| Field | Value |
|-------|-------|
| Severity | INFO |
| AWS API | `network-firewall list-firewalls` |
| Requirement | "Implement segmentation gateways" |

**What it checks:** Verifies AWS Network Firewall is deployed for stateful traffic inspection.

**Why it matters:** Network Firewall provides deep packet inspection and stateful filtering at the VPC level, acting as a segmentation gateway.

**Pass condition:** Network Firewall exists (or INFO if not present)

**Note:** Network Firewall is optional - security groups and NACLs may be sufficient for some environments.

---

### Check 5.3-E: Internet Gateway Placement

| Field | Value |
|-------|-------|
| Severity | INFO |
| AWS API | `ec2 describe-internet-gateways` |
| Requirement | "Define public/private environment segments" |

**What it checks:** Counts Internet Gateways attached to VPCs.

**Why it matters:** IGWs define the boundary between public and private networks. Proper placement ensures only designated public subnets have internet routes.

**Pass condition:** IGWs exist and are attached to VPCs

**Note:** Informational - verify IGW routes only exist in public subnet route tables

## Checks NOT Implemented

| Requirement | Reason |
|-------------|--------|
| SDN proxy/enforcement checks | Requires custom implementation |
| Device attribute-based access | Requires external identity provider integration |
| Behavior-based access decisions | Requires ML/analytics platform |
| Real-time policy enforcement | Requires custom automation |

## External Tools for Full Compliance

To fully implement Activity 5.3.1, integrate AWS with:
- **SIEM** - Splunk, Elastic, AWS Security Lake for log aggregation
- **Network Analytics** - VPC Flow Logs + Athena for traffic analysis
- **Policy Engine** - OPA, custom Lambda for attribute-based decisions
- **IDS/IPS** - AWS Network Firewall, third-party solutions

## How AWS Provides Macro-Segmentation

| ZIG Concept | AWS Equivalent |
|-------------|----------------|
| Public/Private Segments | VPCs with public/private subnets |
| Segmentation Gateways | AWS Network Firewall, NACLs |
| Traffic Monitoring | VPC Flow Logs |
| Egress Control | NAT Gateway, VPC Endpoints |
| Perimeter Security | Internet Gateway, WAF |
