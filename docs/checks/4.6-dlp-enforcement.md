# Activity 4.6.1 - DLP Enforcement Points

## ZIG Requirement Summary

Per NSA ZIG Phase One, Activity 4.6.1 requires:
- Deploy DLP tools at identified enforcement points
- Integrate tagged data attributes with DLP
- Start in "monitor-only" mode, transition to "prevent" mode
- Detect and mitigate data breaches and exfiltration

## Predecessor/Successor Relationships

```
4.3.1 (Classification Tools) → 4.6.1 (Enforcement Points) → 4.6.2 (DLP Part 2)
```

**Predecessor:** Activity 4.3.1 - Implement Data Tagging and Classification Tools
**Successor:** Activity 4.6.2 - DLP Part 2

## AWS Limitations

AWS does not have native DLP. These checks focus on:
- Preventive controls (public access blocks)
- Data protection (Object Lock)
- Network controls (VPC endpoints)
- Macie as a DLP-like discovery tool

Full DLP requires external tools for content inspection and policy enforcement.

## Checks Implemented

---

### Check 4.6-A: S3 Account-Level Public Access Block

| Field | Value |
|-------|-------|
| Severity | HIGH |
| AWS API | `s3control get-public-access-block` |
| Requirement | "Prevent data exfiltration" |

**What it checks:** Verifies all four S3 public access block settings are enabled at the account level.

**Why it matters:** Account-level blocks prevent accidental public exposure of any bucket in the account.

**Pass condition:** All four settings enabled (BlockPublicAcls, IgnorePublicAcls, BlockPublicPolicy, RestrictPublicBuckets)

**Fail condition:** Any setting not enabled

**Remediation:** `aws s3control put-public-access-block --account-id ACCOUNT --public-access-block-configuration BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true`

---

### Check 4.6-B: S3 Bucket-Level Public Access (Compliance Buckets)

| Field | Value |
|-------|-------|
| Severity | BLOCKER |
| AWS API | `s3api get-public-access-block` |
| Requirement | "Protect high-risk data" |

**What it checks:** Verifies compliance-tagged buckets have all public access block settings enabled.

**Why it matters:** Compliance data must never be publicly accessible. This is a critical control.

**Pass condition:** All compliance-tagged buckets have full public access block

**Fail condition:** Any compliance bucket without full public access block

**Remediation:** Enable all public access block settings on compliance buckets immediately

**Note:** Only checks buckets with compliance tags (`DataClassification=critical`, etc.)

---

### Check 4.6-C: S3 Object Lock (Compliance Buckets)

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `s3api get-object-lock-configuration` |
| Requirement | "Protect high-risk data from deletion" |

**What it checks:** Verifies compliance-tagged buckets have Object Lock enabled.

**Why it matters:** Object Lock provides immutability for compliance data, preventing deletion or modification.

**Pass condition:** All compliance-tagged buckets have Object Lock enabled

**Fail condition:** Compliance buckets without Object Lock

**Remediation:** Object Lock must be enabled at bucket creation; may require new bucket and data migration

**Note:** Only checks buckets with compliance tags. Non-compliance buckets are not flagged.

---

### Check 4.6-D: VPC Endpoints for S3

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `ec2 describe-vpc-endpoints` |
| Requirement | "Control data egress paths" |

**What it checks:** Verifies S3 Gateway VPC Endpoints exist.

**Why it matters:** VPC endpoints keep S3 traffic within AWS network, preventing data from traversing the public internet.

**Pass condition:** At least one S3 Gateway Endpoint exists

**Fail condition:** No S3 VPC endpoints configured

**Remediation:** Create S3 Gateway Endpoint in each VPC

---

### Check 4.6-E: Macie Automated Discovery

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `macie2 get-automated-discovery-configuration` |
| Requirement | "Continuous data monitoring" |

**What it checks:** Verifies Macie automated sensitive data discovery is enabled.

**Why it matters:** Automated discovery provides continuous DLP-like monitoring for sensitive data exposure.

**Pass condition:** Macie automated discovery is enabled

**Fail condition:** Automated discovery not enabled (when Macie is active)

**Remediation:** `aws macie2 update-automated-discovery-configuration --status ENABLED`

---

## Compliance Tag Convention

Checks 4.6-B and 4.6-C only apply to buckets with compliance tags:

| Tag Key | Compliance Values |
|---------|-------------------|
| `DataClassification` | `critical`, `sensitive`, `pii`, `phi`, `pci`, `high` |
| `Compliance` | `required`, `critical`, `high` |
| `SecurityLevel` | `critical`, `sensitive`, `high` |

Buckets without these tags are reported informationally but not flagged as failures.

---

## Checks NOT Implemented

| Requirement | Reason |
|-------------|--------|
| Content inspection DLP | Requires external DLP tools |
| Email DLP | Requires email security gateway |
| Endpoint DLP | Requires endpoint agents |
| Network DLP | Requires network inspection tools |
| Monitor → Prevent transition | Requires DLP platform |

## External Tools for Full Compliance

To fully implement Activity 4.6.1, integrate AWS with:
- **DLP** - Symantec, Forcepoint, Microsoft Purview for content inspection
- **CASB** - Netskope, Zscaler for cloud access security
- **Email Security** - Proofpoint, Mimecast for email DLP
- **Endpoint DLP** - CrowdStrike, Carbon Black for endpoint protection

## How AWS Complements External DLP

| AWS Service | Complements |
|-------------|-------------|
| S3 Public Access Block | Preventive control → DLP baseline |
| Macie | Sensitive data discovery → DLP policy input |
| VPC Endpoints | Network control → DLP enforcement point |
| CloudTrail | Audit logging → DLP analytics |
| EventBridge | Real-time events → DLP alerting |
