# Activity 2.4.1 - Deny Device by Default Policy

## ZIG Requirement Summary

Per NSA ZIG Phase One, Activity 2.4.1 requires:
- Block access from all unmanaged remote and local devices
- Managed, compliant devices get risk-based, methodical access
- ACLs, RBAC, ABAC enforcement for device access
- Continuous monitoring of access attempts including denied connections
- Logging and alerting for unapproved device access attempts

## Predecessor/Successor Relationships

```
2.1.2 (NPE & PKI) → 2.4.1 (Deny Device by Default) → None
```

**Predecessor:** Activity 2.1.2 - Device inventory and PKI must be in place before enforcing deny-by-default policies.

## AWS Limitations

AWS APIs can enforce deny-by-default at the network layer (Security Groups, NACLs) but cannot:
- Verify device compliance/posture before granting access (requires NAC/C2C)
- Enforce RBAC/ABAC based on device attributes (requires external policy engine)
- Block unmanaged workstations/laptops (requires UEM/EDM integration)

These checks verify AWS network controls align with deny-by-default principles. Full compliance requires integration with NAC, C2C, and UEM solutions.

## Checks Implemented

---

### Check 2.4.1-A: Default Security Groups Block All Traffic

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `ec2 describe-vpcs`, `ec2 describe-security-groups` |
| Requirement | "Default deny posture" |

**What it checks:** Verifies that default security groups in all VPCs have no ingress rules.

**Why it matters:** Default security groups are automatically assigned to resources if no other SG is specified. If they allow traffic, resources may be unintentionally exposed.

**Pass condition:** All default security groups have zero ingress rules

**Fail condition:** Default security groups contain ingress rules

**Remediation:** Remove all rules from default SGs; create custom SGs with explicit allow rules

---

### Check 2.4.1-B: Security Groups with 0.0.0.0/0 Ingress

| Field | Value |
|-------|-------|
| Severity | HIGH |
| AWS API | `ec2 describe-security-groups` |
| Requirement | "Explicit allow rules only for authorized communications" |

**What it checks:** Identifies security groups allowing inbound traffic from any IP (0.0.0.0/0) on non-web ports.

**Why it matters:** Open ingress rules violate deny-by-default principles. Ports 80/443 are excluded as they're commonly needed for web servers behind load balancers.

**Pass condition:** No security groups allow 0.0.0.0/0 ingress on non-web ports

**Fail condition:** Security groups allow unrestricted ingress on SSH, RDP, databases, etc.

**Remediation:** Restrict ingress to specific IPs/CIDRs; use VPN or bastion hosts for admin access

---

### Check 2.4.1-C: Network ACLs Deny-by-Default

| Field | Value |
|-------|-------|
| Severity | INFO |
| AWS API | `ec2 describe-network-acls` |
| Requirement | "Default deny posture at network layer" |

**What it checks:** Confirms Network ACLs provide implicit deny-all at the end of rule evaluation.

**Why it matters:** NACLs are stateless firewalls at the subnet level. They always have an implicit deny rule (rule 32767) that blocks unmatched traffic.

**Pass condition:** Always passes - NACLs inherently deny by default

**Note:** Consider adding explicit deny rules for known-bad IP ranges for defense in depth

---

### Check 2.4.1-D: VPC Flow Logs Enabled

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `ec2 describe-flow-logs` |
| Requirement | "Traffic logging for visibility and forensics" |

**What it checks:** Verifies all VPCs have flow logs enabled.

**Why it matters:** Flow logs capture network traffic metadata for security analysis, troubleshooting, and compliance. Without them, network activity is invisible.

**Pass condition:** All VPCs have at least one flow log configured

**Fail condition:** VPCs exist without flow logs

**Remediation:** Enable VPC Flow Logs to CloudWatch Logs or S3

| Variable | Description |
|----------|-------------|

## Checks NOT Implemented

| Requirement | Reason |
|-------------|--------|
| Device posture/compliance verification | Requires NAC/C2C integration |
| RBAC/ABAC based on device attributes | Requires external policy engine |
| Unmanaged workstation blocking | Requires UEM/EDM solution |
| Network Access Control (NAC) | External tooling required |
| Exception management and tracking | Process artifact - not API queryable |

## External Tools for Full Compliance

To fully implement Activity 2.4.1, integrate AWS controls with:
- **NAC solution** - For device posture checks before network access
- **C2C (Comply-to-Connect)** - For compliance-based network authorization
- **UEM/EDM** - For workstation/laptop/mobile device management
- **SIEM** - For centralized logging and alerting on denied access attempts
