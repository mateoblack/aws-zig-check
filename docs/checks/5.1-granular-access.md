# Activity 5.1.2 - Granular Control Access Rules

## ZIG Requirement Summary

Per NSA ZIG Phase One, Activity 5.1.2 requires:
- Define data tagging filters for API infrastructure
- Enforce authentication for all APIs at the API layer
- Integrate API decision points with SDN architecture
- Enable secure API communications (HTTPS, mTLS)

## Predecessor/Successor Relationships

```
5.1.1 (Granular Rules Part 1) → 5.1.2 (Granular Rules Part 2) → None
```

**Predecessor:** Activity 5.1.1 - Define Granular Control Access Rules Part 1
**Successor:** None

## AWS Limitations

AWS provides API-level controls but not full SDN programmability:
- API Gateway provides authentication/authorization
- WAF provides web application filtering
- No native API data tagging filters
- SDN is abstracted by AWS

## Checks Implemented

---

### Check 5.1-A: WAF WebACLs Configured

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `wafv2 list-web-acls` |
| Requirement | "API layer protection" |

**What it checks:** Verifies WAF WebACLs are configured for regional and CloudFront resources.

**Why it matters:** WAF provides application-layer filtering for APIs and web applications, blocking common attacks.

**Pass condition:** At least one WAF WebACL exists

**Fail condition:** No WAF WebACLs configured

**Remediation:** Create WAF WebACL and associate with API Gateway, ALB, or CloudFront

---

### Check 5.1-B: Route 53 DNS Firewall

| Field | Value |
|-------|-------|
| Severity | INFO |
| AWS API | `route53resolver list-firewall-rule-groups` |
| Requirement | "Network-level access control" |

**What it checks:** Verifies Route 53 Resolver DNS Firewall rule groups exist.

**Why it matters:** DNS Firewall can block DNS queries to malicious domains, providing network-level protection.

**Pass condition:** DNS Firewall rule groups exist (or INFO if not configured)

**Note:** This is optional - not all environments need DNS-level filtering.

---

### Check 5.1-C: VPC Endpoints (PrivateLink)

| Field | Value |
|-------|-------|
| Severity | LOW |
| AWS API | `ec2 describe-vpc-endpoints` |
| Requirement | "Secure API communications" |

**What it checks:** Verifies VPC Endpoints exist for AWS service access.

**Why it matters:** VPC Endpoints keep AWS API traffic within the AWS network, avoiding public internet.

**Pass condition:** At least one VPC Endpoint exists

**Fail condition:** No VPC Endpoints configured

**Remediation:** Create VPC Endpoints for frequently used AWS services (S3, DynamoDB, etc.)

---

## Checks NOT Implemented

| Requirement | Reason |
|-------------|--------|
| API data tagging filters | Requires custom implementation |
| SDN API decision points | AWS abstracts SDN |
| mTLS enforcement | Application-level configuration |
| API gateway integration | Covered in Pillar 3 (Application) |

## External Tools for Full Compliance

To fully implement Activity 5.1.2, integrate AWS with:
- **API Gateway** - Kong, Apigee for advanced API management
- **Service Mesh** - Istio, AWS App Mesh for mTLS
- **Policy Engine** - OPA for API policy decisions
