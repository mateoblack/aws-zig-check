# Activity 4.5.1 - Data Rights Management & Protection

## ZIG Requirement Summary

Per NSA ZIG Phase One, Activity 4.5.1 requires:
- Implement DRM and protection solutions for high-risk data
- Enable encryption for data at rest and in transit
- Deploy Key Management System (KMS) for encryption keys
- Apply fine-grained permissions to high-risk data assets

## Predecessor/Successor Relationships

```
4.2.2 (Interoperability Standards) → 4.5.1 (DRM Part 1) → 4.5.2 (DRM Part 2)
```

**Predecessor:** Activity 4.2.2 - Interoperability Standards
**Successor:** Activity 4.5.2 - DRM and Protection Tools Part 2

## AWS Limitations

AWS provides strong encryption but limited DRM:
- No true Digital Rights Management (document-level controls)
- Encryption is storage-level, not content-aware
- No watermarking or usage tracking
- Requires external DRM tools for full compliance

## Checks Implemented

---

### Check 4.5-A: S3 Default Encryption

| Field | Value |
|-------|-------|
| Severity | HIGH (compliance buckets) / MEDIUM (others) |
| AWS API | `s3api get-bucket-encryption` |
| Requirement | "Encrypt sensitive data at rest" |

**What it checks:** Verifies S3 buckets have default encryption enabled (SSE-S3 or SSE-KMS).

**Why it matters:** Unencrypted buckets expose data if access controls fail. Encryption is defense-in-depth.

**Pass condition:** All buckets have default encryption enabled

**Fail condition:** Buckets without default encryption (compliance-tagged = HIGH, others = MEDIUM)

**Remediation:** `aws s3api put-bucket-encryption --bucket BUCKET --server-side-encryption-configuration ...`

---

### Check 4.5-B: S3 Bucket Key

| Field | Value |
|-------|-------|
| Severity | LOW |
| AWS API | `s3api get-bucket-encryption` |
| Requirement | "Efficient key management" |

**What it checks:** Verifies KMS-encrypted buckets have S3 Bucket Key enabled.

**Why it matters:** Bucket Key reduces KMS API calls and costs while maintaining encryption.

**Pass condition:** All KMS-encrypted buckets have Bucket Key enabled

**Fail condition:** KMS buckets without Bucket Key

**Remediation:** Enable Bucket Key in bucket encryption configuration

---

### Check 4.5-C: EBS Default Encryption

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `ec2 get-ebs-encryption-by-default` |
| Requirement | "Encrypt data at rest" |

**What it checks:** Verifies EBS default encryption is enabled at the account/region level.

**Why it matters:** Default encryption ensures all new EBS volumes are automatically encrypted.

**Pass condition:** EBS default encryption is enabled

**Fail condition:** EBS default encryption not enabled

**Remediation:** `aws ec2 enable-ebs-encryption-by-default`

---

### Check 4.5-D: RDS Encryption

| Field | Value |
|-------|-------|
| Severity | HIGH |
| AWS API | `rds describe-db-instances` |
| Requirement | "Encrypt databases" |

**What it checks:** Verifies all RDS instances have storage encryption enabled.

**Why it matters:** Database encryption protects sensitive data at rest from unauthorized access.

**Pass condition:** All RDS instances are encrypted

**Fail condition:** Unencrypted RDS instances exist

**Remediation:** Enable encryption (requires snapshot and restore for existing instances)

---

### Check 4.5-E: KMS Key Rotation

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `kms list-keys`, `kms get-key-rotation-status` |
| Requirement | "Key management" |

**What it checks:** Verifies customer-managed KMS keys have automatic rotation enabled.

**Why it matters:** Key rotation limits the impact of key compromise and meets compliance requirements.

**Pass condition:** All customer-managed keys have rotation enabled

**Fail condition:** Keys without automatic rotation

**Remediation:** `aws kms enable-key-rotation --key-id KEY_ID`

---

### Check 4.5-F: Secrets Manager Rotation

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `secretsmanager list-secrets` |
| Requirement | "Protect credentials" |

**What it checks:** Verifies Secrets Manager secrets have automatic rotation configured.

**Why it matters:** Automatic rotation reduces risk of credential compromise and ensures freshness.

**Pass condition:** All secrets have rotation enabled

**Fail condition:** Secrets without rotation configured

**Remediation:** Configure rotation Lambda function for each secret

---

## Checks NOT Implemented

| Requirement | Reason |
|-------------|--------|
| Digital Rights Management | Requires external DRM tools (Microsoft Purview, etc.) |
| Document-level encryption | Requires application-level implementation |
| Watermarking | Requires external tools |
| Usage tracking | Requires DRM platform |
| Data-in-use encryption | Requires specialized compute (Nitro Enclaves) |

## External Tools for Full Compliance

To fully implement Activity 4.5.1, integrate AWS with:
- **DRM** - Microsoft Purview, Adobe Experience Manager for document rights
- **Key Management** - HashiCorp Vault for multi-cloud key management
- **Encryption** - AWS Nitro Enclaves for data-in-use protection
- **Certificate Management** - ACM, external PKI for TLS certificates

## How AWS Complements External DRM

| AWS Service | Complements |
|-------------|-------------|
| KMS | Key management → DRM key storage |
| S3 Encryption | Storage encryption → DRM baseline |
| Secrets Manager | Credential management → DRM authentication |
| IAM | Access control → DRM authorization |
| CloudHSM | Hardware key storage → high-security DRM |
