# Activity 6.1.2 - Organization Access Profile

## ZIG Requirement Summary

Per NSA ZIG Phase One, Activity 6.1.2 requires:
- Develop access profile rules for mission/task and non-mission/task DAAS access
- Use data from User, Data, Network & Environment, and Device pillars
- Employ deny-by-default strategy
- Create component mission-/task-critical profile rules

## Predecessor/Successor Relationships

```
None → 6.1.2 (Organization Access Profile) → 6.1.3
```

**Predecessor:** None
**Successor:** Activity 6.1.3 - Enterprise Security Profile Part 1

## AWS Limitations

AWS provides access control through IAM but doesn't have a unified "access profile" concept:
- IAM policies define permissions
- SCPs provide organization-wide guardrails
- Permission boundaries limit maximum permissions
- Access Analyzer identifies external access

Full ZIG compliance requires:
- Documented access profiles per role
- Regular access reviews
- Integration with HR/identity systems

## Checks Implemented

---

### Check 6.1-A: IAM Access Analyzer Enabled

| Field | Value |
|-------|-------|
| Severity | HIGH |
| AWS API | `accessanalyzer list-analyzers` |
| Requirement | "Verify and validate ACLs are operational" |

**What it checks:** Verifies IAM Access Analyzer is enabled to identify resources shared externally.

**Why it matters:** Access Analyzer continuously monitors for resources accessible outside your account, supporting the ZIG requirement to validate access controls.

**Pass condition:** At least one active Access Analyzer exists

**Fail condition:** No Access Analyzer enabled

**Remediation:** Enable IAM Access Analyzer in the IAM console or via CLI

---

### Check 6.1-B: Service Control Policies (SCPs)

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `organizations list-policies` |
| Requirement | "Deny-by-default strategy" |

**What it checks:** Verifies SCPs are configured beyond the default FullAWSAccess policy.

**Why it matters:** SCPs provide organization-wide guardrails that enforce access restrictions across all accounts, supporting deny-by-default.

**Pass condition:** More than 1 SCP exists (beyond default)

**Fail condition:** Only default SCP in place

**Note:** Only applicable if account is part of AWS Organizations

**Remediation:** Create SCPs to restrict dangerous actions organization-wide

---

### Check 6.1-C: Permission Boundaries

| Field | Value |
|-------|-------|
| Severity | INFO |
| AWS API | `iam list-users`, `iam list-roles` |
| Requirement | "Define levels of access required for each role" |

**What it checks:** Counts IAM users and roles with permission boundaries configured.

**Why it matters:** Permission boundaries set maximum permissions, preventing privilege escalation even if policies are misconfigured.

**Pass condition:** Permission boundaries in use (informational)

**Note:** Optional but recommended for delegated administration

**Remediation:** Attach permission boundaries to limit maximum permissions

---

### Check 6.1-D: Overly Permissive Policies

| Field | Value |
|-------|-------|
| Severity | HIGH |
| AWS API | `iam list-policies`, `iam get-policy-version` |
| Requirement | "Least privilege" |

**What it checks:** Identifies customer-managed policies with Action:* and Resource:*.

**Why it matters:** Policies allowing all actions on all resources violate least privilege and deny-by-default principles.

**Pass condition:** No policies with Action:* Resource:*

**Fail condition:** Overly permissive policies exist

**Remediation:** Scope policies to specific actions and resources needed

---

## Checks NOT Implemented

| Requirement | Reason |
|-------------|--------|
| Documented access profiles | Requires external documentation |
| Role-based access mapping | Requires HR/identity integration |
| Time-based access constraints | Requires custom implementation |
| Regular access reviews | Requires process, not just tooling |

## External Tools for Full Compliance

To fully implement Activity 6.1.2:
- **Identity Governance** - SailPoint, Saviynt for access reviews
- **Documentation** - Confluence, SharePoint for access profile docs
- **SIEM** - Splunk, Elastic for access pattern analysis

## How AWS Provides Access Profiles

| ZIG Concept | AWS Equivalent |
|-------------|----------------|
| Access Profile Rules | IAM Policies |
| Deny-by-Default | Implicit deny, SCPs |
| Role-Based Access | IAM Roles |
| Access Validation | IAM Access Analyzer |
| Permission Limits | Permission Boundaries |
