# Activity 4.4.3 - File Activity Monitoring

## ZIG Requirement Summary

Per NSA ZIG Phase One, Activity 4.4.3 requires:
- Monitor critical data classifications in applications, services, and repositories
- Feed analytics into SIEM with basic data attributes
- Establish business rules for critical data designations
- Integrate with monitoring systems (SIEM, XDR)

## Predecessor/Successor Relationships

```
None → 4.4.3 (File Activity Monitoring Part 1) → 4.4.4 (File Activity Monitoring Part 2)
```

**Predecessor:** None (but integrates with 4.2.1, 4.3.1, 4.6.1)
**Successor:** Activity 4.4.4 - File Activity Monitoring Part 2

## AWS Limitations

AWS provides logging but not true File Activity Monitoring (FAM):
- CloudTrail logs API calls, not file-level access patterns
- S3 access logs are basic (no content inspection)
- No built-in UEBA or behavioral analytics
- Requires external FAM/DLP tools for full compliance

## Checks Implemented

---

### Check 4.4-A: CloudTrail S3 Data Events

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `cloudtrail list-trails`, `cloudtrail get-event-selectors` |
| Requirement | "Monitor critical data in repositories" |

**What it checks:** Verifies at least one CloudTrail trail is logging S3 data events (object-level operations).

**Why it matters:** S3 data events capture GetObject, PutObject, DeleteObject - essential for understanding data access patterns.

**Pass condition:** At least one trail has S3 data events enabled

**Fail condition:** No trails logging S3 data events

**Remediation:** Enable S3 data events on CloudTrail trail

---

### Check 4.4-B: S3 Server Access Logging (Compliance Buckets)

| Field | Value |
|-------|-------|
| Severity | HIGH |
| AWS API | `s3api get-bucket-logging` |
| Requirement | "Monitor critical data classifications" |

**What it checks:** Verifies compliance-tagged S3 buckets have server access logging enabled.

**Why it matters:** Access logs provide detailed records of requests made to compliance-critical buckets.

**Pass condition:** All compliance-tagged buckets have access logging enabled

**Fail condition:** Compliance-tagged buckets without access logging

**Remediation:** Enable server access logging for compliance buckets

**Note:** Only checks buckets with compliance tags (`DataClassification=critical`, etc.)

---

### Check 4.4-C: CloudWatch Log Groups for Data Access

| Field | Value |
|-------|-------|
| Severity | LOW |
| AWS API | `logs describe-log-groups` |
| Requirement | "Integration with monitoring system" |

**What it checks:** Looks for CloudWatch log groups related to data access (names containing 'data', 's3', 'rds').

**Why it matters:** Centralized logging enables SIEM integration and analytics on data access patterns.

**Pass condition:** Log groups for data monitoring exist

**Fail condition:** No data-related log groups found

**Remediation:** Configure CloudWatch Logs for data access monitoring

---

## Checks NOT Implemented

| Requirement | Reason |
|-------------|--------|
| File Activity Monitoring (FAM) | Requires external tools (Varonis, etc.) |
| UEBA/behavioral analytics | Requires external UEBA platform |
| Content inspection | Requires DLP tools |
| Real-time alerting on data access | Requires SIEM integration |
| Critical data designation rules | Organizational process |

## External Tools for Full Compliance

To fully implement Activity 4.4.3, integrate AWS with:
- **FAM** - Varonis, Netwrix for file activity monitoring
- **SIEM** - Splunk, Elastic, AWS Security Lake for log aggregation
- **UEBA** - Exabeam, Securonix for behavioral analytics
- **DLP** - Symantec, Forcepoint for content inspection

## How AWS Complements External FAM

| AWS Service | Complements |
|-------------|-------------|
| CloudTrail | API-level logging → SIEM ingestion |
| S3 Access Logs | Request-level logging → FAM enrichment |
| CloudWatch Logs | Centralized logs → SIEM integration |
| Security Lake | OCSF-normalized logs → security analytics |
| EventBridge | Real-time events → alerting workflows |
