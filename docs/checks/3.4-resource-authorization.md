# Activity 3.4.1/3.4.3 - Resource Authorization

## ZIG Requirement Summary

Per NSA ZIG Phase One, Activities 3.4.1 and 3.4.3 require:
- Deploy approved resource authorization gateways for external-facing applications
- Integrate access/authorization with identities and devices
- Establish Software-Defined Compute (SDC) standards
- Identify applications that can support SDC approach
- Implement programmatic resource authorization

## AWS Limitations

AWS provides authorization primitives but not complete resource authorization gateways:
- API Gateway provides API-level authorization
- ALB provides authentication but limited authorization
- No unified authorization gateway across all services
- SDC requires IaC adoption and process changes

## Checks Implemented

---

### Check 3.4-A: API Gateway Authorization

| Field | Value |
|-------|-------|
| Severity | HIGH |
| AWS API | `apigateway get-rest-apis`, `apigateway get-authorizers`, `apigatewayv2 get-apis` |
| Requirement | "Deploy resource authorization gateways for external-facing applications" |

**What it checks:** Verifies all API Gateway APIs (REST and HTTP) have authorizers configured.

**Why it matters:** APIs without authorization allow unauthenticated access to backend services, violating Zero Trust principles.

**Pass condition:** All API Gateway APIs have at least one authorizer

**Fail condition:** APIs exist without authorizers configured

**Remediation:** Configure Lambda authorizers, Cognito User Pools, or IAM authorization

---

### Check 3.4-B: ALB Authentication

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `elbv2 describe-load-balancers`, `elbv2 describe-listeners`, `elbv2 describe-rules` |
| Requirement | "Integrate authorization with identities" |

**What it checks:** Verifies Application Load Balancers have authentication actions (Cognito or OIDC).

**Why it matters:** ALB authentication provides a centralized authentication layer for web applications before traffic reaches backends.

**Pass condition:** All ALBs have authenticate-cognito or authenticate-oidc actions

**Fail condition:** ALBs without authentication configured

**Remediation:** Configure ALB authentication with Cognito or OIDC provider

---

### Check 3.4-C: Lambda Resource Policies

| Field | Value |
|-------|-------|
| Severity | HIGH |
| AWS API | `lambda list-functions`, `lambda get-policy` |
| Requirement | "Resource authorization" |

**What it checks:** Identifies Lambda functions with overly permissive resource policies (Principal: "*" without conditions).

**Why it matters:** Lambda functions with public access can be invoked by anyone, bypassing authorization controls.

**Pass condition:** No Lambda functions with unrestricted Principal: "*"

**Fail condition:** Lambda functions with Principal: "*" and no conditions

**Remediation:** Add conditions to resource policies or use API Gateway for access control

---

### Check 3.4-D: EKS OIDC Provider

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `eks list-clusters`, `eks describe-cluster` |
| Requirement | "Programmatic resource authorization" |

**What it checks:** Verifies EKS clusters have OIDC provider configured for IAM Roles for Service Accounts (IRSA).

**Why it matters:** IRSA enables fine-grained IAM permissions for Kubernetes pods, implementing least privilege for workloads.

**Pass condition:** All EKS clusters have OIDC issuer configured

**Fail condition:** EKS clusters without OIDC provider

**Remediation:** Enable OIDC provider: `eksctl utils associate-iam-oidc-provider --cluster CLUSTER`

---

### Check 3.4-E: Infrastructure as Code (SDC)

| Field | Value |
|-------|-------|
| Severity | LOW |
| AWS API | `cloudformation list-stack-sets`, `cloudformation list-stacks` |
| Requirement | "Software-Defined Compute standards" |

**What it checks:** Verifies CloudFormation stacks or stacksets exist, indicating IaC adoption.

**Why it matters:** SDC requires infrastructure defined as code for programmatic management, version control, and compliance.

**Pass condition:** CloudFormation stacks or stacksets exist

**Fail condition:** No CloudFormation resources found

**Remediation:** Adopt IaC with CloudFormation or Terraform for SDC approach

---

### Check 3.4-F: Service Control Policies

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `organizations list-policies`, `organizations describe-organization` |
| Requirement | "Enterprise sets standards on policy enforcement" |

**What it checks:** Verifies Service Control Policies exist for organization-level resource authorization.

**Why it matters:** SCPs provide guardrails across all accounts, enforcing organization-wide authorization policies.

**Pass condition:** Custom SCPs configured in AWS Organization

**Fail condition:** AWS Organization exists but no custom SCPs

**Remediation:** Create SCPs for organization-wide resource authorization

## Checks NOT Implemented

|-------------|--------|
| Unified authorization gateway | AWS has no single gateway product |

| Exception management | Process - not API queryable |
| CI/CD pipeline integration | Requires pipeline inspection |
| Attribute enrichment | Requires custom implementation |

## External Tools for Full Compliance

To fully implement Activities 3.4.1/3.4.3, integrate AWS with:
- **Authorization Gateway** - Kong, Apigee, or custom PEP/PDP
- **Policy Engine** - Open Policy Agent (OPA), Cedar for fine-grained authorization
- **IaC Tools** - Terraform, Pulumi for multi-cloud SDC
- **GitOps** - ArgoCD, Flux for declarative infrastructure
- **ABAC/RBAC** - External policy management systems

## How AWS Complements External Authorization

| AWS Service | Complements |
|-------------|-------------|
| API Gateway | API-level authorization → external policy engine |
| ALB | Authentication layer → external IdP |
| IAM | AWS authorization → federated with external IdP |
| CloudFormation | IaC → GitOps workflows |
| Organizations/SCPs | Guardrails → enterprise policy framework |
