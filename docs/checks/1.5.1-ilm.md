# Activity 1.5.1 - Identity Lifecycle Management (ILM)

## ZIG Requirement Summary

Per NSA ZIG Phase One, Capability 1.5 requires:
- Standardized Identity Lifecycle Management (ILM) processes
- Single process for issuing, updating, and revoking credentials
- Automatic deactivation of credentials when users leave or roles change
- Identity federation for cross-domain authentication
- SSO capability for authenticated users

## Checks Implemented

### Check 1.5.1-A: Inactive IAM Users (No Console Login in 90+ Days)

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `iam get-credential-report` |
| Fields Checked | `password_enabled`, `password_last_used` |

**What it checks:** Identifies IAM users with console access who have not logged in for 90+ days.

**Why it matters:** ZIG states credentials should be "deactivated when Users/PEs leave or their roles change." Users who haven't logged in for extended periods may have:
- Left the organization
- Changed roles and no longer need access
- Forgotten accounts that increase attack surface

Active ILM processes should identify and deprovision these accounts.

**Pass condition:** No IAM users with console access inactive for 90+ days

**Fail condition:** One or more IAM users have not used their password in 90+ days

**Remediation:**
```bash
# Review inactive users
aws iam get-credential-report --query 'Content' --output text | base64 -d | \
  awk -F, 'NR>1 && $4=="true" {print $1, $5}'

# Disable console access for inactive user
aws iam delete-login-profile --user-name <username>

# Or delete user entirely if no longer needed
aws iam delete-user --user-name <username>
```

### Check 1.5.1-B: Inactive Access Keys (No Use in 90+ Days)

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `iam get-credential-report` |
| Fields Checked | `access_key_1_active`, `access_key_1_last_used_date`, `access_key_2_*` |

**What it checks:** Identifies active access keys that have not been used in 90+ days.

**Why it matters:** Unused access keys indicate:
- Service/application no longer in use
- User no longer needs programmatic access
- Orphaned credentials that could be compromised

ILM processes should identify and revoke unused credentials.

**Pass condition:** No active access keys unused for 90+ days

**Fail condition:** One or more active access keys have not been used in 90+ days

**Remediation:**
```bash
# List access keys and last used date
aws iam list-access-keys --user-name <username>
aws iam get-access-key-last-used --access-key-id <key-id>

# Deactivate unused key
aws iam update-access-key --user-name <username> --access-key-id <key-id> --status Inactive

# Delete unused key
aws iam delete-access-key --user-name <username> --access-key-id <key-id>
```

### Check 1.5.1-C: Access Keys Never Used

| Field | Value |
|-------|-------|
| Severity | LOW |
| AWS API | `iam get-credential-report` |
| Fields Checked | `access_key_*_active`, `access_key_*_last_used_date` |

**What it checks:** Identifies active access keys that have never been used since creation.

**Why it matters:** Keys created but never used may indicate:
- Abandoned setup/configuration
- Keys created "just in case" (violates least privilege)
- Forgotten credentials

These should be reviewed and deleted if not needed.

**Pass condition:** No active access keys that have never been used

**Fail condition:** One or more active access keys have never been used

**Note:** New keys (< 7 days old) are excluded from this check to allow for setup time.

**Remediation:**
```bash
# Delete unused key
aws iam delete-access-key --user-name <username> --access-key-id <key-id>
```

### Check 1.5.1-D: IAM Users Without Group Membership

| Field | Value |
|-------|-------|
| Severity | LOW |
| AWS API | `iam list-users`, `iam list-groups-for-user` |
| Fields Checked | Group membership count |

**What it checks:** Identifies IAM users that are not members of any IAM group.

**Why it matters:** Proper ILM uses groups to manage permissions:
- Groups represent roles/job functions
- Permissions are assigned to groups, not individual users
- When users change roles, group membership changes
- Easier to audit and manage at scale

Users without group membership may have:
- Direct policy attachments (harder to manage)
- Been orphaned during role changes
- Inconsistent permission management

**Pass condition:** All IAM users belong to at least one group

**Fail condition:** One or more IAM users have no group membership

**Note:** This is LOW severity because some organizations use direct policy attachment intentionally. However, group-based management is a best practice for ILM.

**Remediation:**
```bash
# List user's groups
aws iam list-groups-for-user --user-name <username>

# Add user to appropriate group
aws iam add-user-to-group --user-name <username> --group-name <groupname>

# Review and remove direct policy attachments
aws iam list-attached-user-policies --user-name <username>
aws iam detach-user-policy --user-name <username> --policy-arn <policy-arn>
```
