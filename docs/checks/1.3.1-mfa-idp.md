# Activity 1.3.1 - Multi-Factor Authentication (MFA) and Identity Provider (IdP)

## ZIG Requirement Summary

Per NSA ZIG Phase One, Capability 1.3 requires:
- Centralized MFA provider integrated with Identity Provider (IdP)
- Retirement of local and/or built-in accounts
- All users must authenticate with at least 2 of 3 factors: knowledge, possession, or inherence
- Phased rollout starting with standard users, then privileged users

## Checks Implemented

### Check 1.3.1-A: Root Account MFA

| Field | Value |
|-------|-------|
| Severity | BLOCKER |
| AWS API | `iam get-account-summary` |
| Field Checked | `SummaryMap.AccountMFAEnabled` |

**What it checks:** Whether the AWS root account has any MFA device enabled.

**Why it matters:** The root account has unrestricted access to all AWS resources. Without MFA, a compromised root password gives an attacker complete control. This is the single most critical account to protect.

**Pass condition:** `AccountMFAEnabled == 1`

**Fail condition:** `AccountMFAEnabled == 0`

**Remediation:** 
```bash
# Enable virtual MFA on root (must be done via Console)
# 1. Sign in as root to AWS Console
# 2. Go to IAM → Security credentials
# 3. Assign MFA device
```

---

### Check 1.3.1-B: IAM Users Console Access Without MFA

| Field | Value |
|-------|-------|
| Severity | HIGH |
| AWS API | `iam generate-credential-report`, `iam get-credential-report` |
| Fields Checked | `password_enabled`, `mfa_active` |

**What it checks:** Identifies IAM users who have console access (password enabled) but do not have MFA configured.

**Why it matters:** Users with passwords but no MFA can be compromised via phishing or credential stuffing. ZIG requires 2-factor authentication for all users.

**Pass condition:** All IAM users with `password_enabled=true` also have `mfa_active=true`

**Fail condition:** Any IAM user has `password_enabled=true` AND `mfa_active=false`

**Remediation:**
```bash
# List users without MFA
aws iam list-users --query 'Users[].UserName'

# For each user, they must enable MFA via Console or CLI
aws iam enable-mfa-device --user-name <user> --serial-number <mfa-arn> --authentication-code1 <code1> --authentication-code2 <code2>
```

---

### Check 1.3.1-C: IAM Identity Center (SSO) Configured

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `sso-admin list-instances` |
| Field Checked | `Instances[].InstanceArn` |

**What it checks:** Whether AWS IAM Identity Center (formerly AWS SSO) is enabled as a centralized Identity Provider.

**Why it matters:** ZIG emphasizes centralized identity management. IAM Identity Center provides:
- Single sign-on across AWS accounts
- Centralized MFA enforcement
- Integration with external IdPs (Okta, Azure AD, etc.)
- Easier account lifecycle management

**Pass condition:** At least one IAM Identity Center instance exists

**Fail condition:** No IAM Identity Center instances found

**Remediation:**
```bash
# Enable IAM Identity Center via Console
# 1. Go to IAM Identity Center in AWS Console
# 2. Click "Enable"
# 3. Choose identity source (Identity Center directory or external IdP)
```

---

### Check 1.3.1-D: Local IAM Users When SSO Exists

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `sso-admin list-instances`, `iam get-credential-report` |
| Fields Checked | SSO instance existence, IAM users with `password_enabled=true` |

**What it checks:** If IAM Identity Center is configured, checks whether local IAM users with console passwords still exist.

**Why it matters:** ZIG states "Retirement of local and/or built-in accounts and groups is a critical piece to this capability." Once a centralized IdP is in place, local IAM users should be migrated and retired.

**Pass condition:** SSO is configured AND no IAM users have console passwords, OR SSO is not configured (check N/A)

**Fail condition:** SSO is configured AND IAM users with console passwords exist

**Note:** Some organizations intentionally keep 1-2 break-glass IAM users. This finding should be reviewed - if break-glass accounts are intentional and properly secured (MFA, limited use), document the exception.

**Remediation:**
```bash
# Identify IAM users with console access
aws iam get-credential-report --query 'Content' --output text | base64 -d | grep ",true," 

# For each user:
# 1. Create equivalent user in IAM Identity Center
# 2. Migrate permissions via Permission Sets
# 3. Delete IAM user console password: aws iam delete-login-profile --user-name <user>
# 4. Or delete user entirely: aws iam delete-user --user-name <user>
```

### Check 1.3.1-E: MFA Enforcement SCP

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `organizations describe-organization`, `organizations list-policies` |
| Fields Checked | Organization exists, SCPs with MFA deny conditions |

**What it checks:** Whether an AWS Organizations Service Control Policy (SCP) exists that denies actions when MFA is not present.

**Why it matters:** Individual IAM policies can be bypassed or misconfigured. An SCP provides organization-wide enforcement that cannot be overridden by member accounts. This is defense-in-depth for MFA requirements.

**Pass condition:** Organization exists AND at least one SCP contains MFA condition (`aws:MultiFactorAuthPresent` or `aws:MultiFactorAuthAge`)

**Fail condition:** No organization, OR organization exists but no MFA-enforcing SCPs found

**Example MFA Enforcement SCP:**
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyAllExceptListedIfNoMFA",
      "Effect": "Deny",
      "NotAction": [
        "iam:CreateVirtualMFADevice",
        "iam:EnableMFADevice",
        "iam:GetUser",
        "iam:ListMFADevices",
        "iam:ListVirtualMFADevices",
        "iam:ResyncMFADevice",
        "sts:GetSessionToken"
      ],
      "Resource": "*",
      "Condition": {
        "BoolIfExists": {
          "aws:MultiFactorAuthPresent": "false"
        }
      }
    }
  ]
}
```

**Remediation:**
```bash
# Create MFA enforcement SCP (requires management account access)
aws organizations create-policy \
  --name "RequireMFA" \
  --type SERVICE_CONTROL_POLICY \
  --content file://mfa-scp.json

# Attach to root or specific OUs
aws organizations attach-policy \
  --policy-id <policy-id> \
  --target-id <root-id-or-ou-id>
```

### Check 1.3.1-F: External IdP Federation (CAC/PIV/Enterprise PKI)

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `iam list-saml-providers`, `iam list-open-id-connect-providers` |
| Fields Checked | SAML provider ARNs, OIDC provider ARNs |

**What it checks:** Whether external Identity Providers are federated into AWS via SAML or OIDC.

**Why it matters:** ZIG Activity 1.3.1 specifies:
- "DoW PKI Multi-Factor Authentication (e.g., CAC, DPIV, DoW Issued PIV-I, FIPS 201 Compliant softcerts)"
- "Enterprise IdP/MFA solution"

CAC (Common Access Card), PIV (Personal Identity Verification), and other PKI-based authentication cannot be done natively in AWS. These require an external Identity Provider (like Okta, Azure AD, ICAM, or a DoD IdP) that:
1. Handles the PKI/smartcard authentication
2. Federates the identity into AWS via SAML or OIDC

Without external IdP federation, the organization cannot meet the ZIG requirement for enterprise PKI authentication.

**Pass condition:** At least one SAML or OIDC provider is configured in IAM

**Fail condition:** No SAML or OIDC providers found

**Note:** This check verifies federation exists but cannot verify the external IdP is configured for CAC/PIV - that configuration is outside AWS.

**Remediation:**
```bash
# List existing SAML providers
aws iam list-saml-providers

# List existing OIDC providers  
aws iam list-open-id-connect-providers

# To add a SAML provider (requires IdP metadata XML):
aws iam create-saml-provider \
  --saml-metadata-document file://idp-metadata.xml \
  --name MyEnterpriseIdP

# To add an OIDC provider:
aws iam create-open-id-connect-provider \
  --url https://idp.example.com \
  --client-id-list my-client-id \
  --thumbprint-list <certificate-thumbprint>
```

**For IAM Identity Center with External IdP:**
1. Go to IAM Identity Center in AWS Console
2. Settings → Identity source → Change identity source
3. Choose "External identity provider"
4. Configure SAML 2.0 with your enterprise IdP

## Checks NOT Implemented

| Check | Reason |
|-------|--------|
| Biometric/inherence factors | AWS does not expose biometric MFA details via API |
| External IdP MFA policies | Okta, Azure AD, etc. policies are outside AWS API scope |
| IAM Identity Center MFA settings | Requires Identity Store API which has limited MFA config visibility |
| User education/rollout status | Not queryable via API |
| Hardware vs Virtual MFA distinction | Per requirements, any MFA satisfies the check |
