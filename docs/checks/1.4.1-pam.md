# Activity 1.4.1 - Privileged Access Management (PAM)

## ZIG Requirement Summary

Per NSA ZIG Phase One, Capability 1.4 requires:
- Removal of permanent administrator/elevated privileges
- Migration of privileged users to a centralized PAM solution
- Just-In-Time (JIT) access for administrative tasks
- Password vault for privileged accounts
- Integration with security analytics for anomaly detection

## Important Note: External PAM Required

**Full PAM implementation requires external tooling.** AWS does not provide a native PAM solution with:
- JIT approval workflows
- Password vaulting with check-out/check-in
- Session recording
- Privilege escalation approvals


**The checks below verify AWS is configured in a way that supports PAM principles**, but do not validate a complete PAM implementation.

---

## Checks Implemented

### Check 1.4.1-A: IAM Users with Permanent Admin Access

| Field | Value |
|-------|-------|
| Severity | HIGH |
| AWS API | `iam list-users`, `iam list-attached-user-policies`, `iam list-user-policies`, `iam list-groups-for-user` |
| Fields Checked | Attached policies containing AdministratorAccess or admin-equivalent |

**What it checks:** Identifies IAM users who have permanent administrative privileges attached directly or via groups.

**Why it matters:** ZIG states "Permanent elevated privileges are removed, and Users/PEs are required to request Just-In-Time (JIT) access for administrative tasks." IAM users with permanent admin access violate this principle.

**Pass condition:** No IAM users have AdministratorAccess or equivalent policies attached

**Fail condition:** One or more IAM users have permanent admin privileges

**Note:** Some organizations keep 1-2 break-glass admin users. If intentional, document the exception and ensure:
- MFA is enabled
- Access keys are disabled or rotated frequently
- Usage is monitored and alerted

**Remediation:**
```bash
# List users with AdministratorAccess
for user in $(aws iam list-users --query 'Users[].UserName' --output text); do
  aws iam list-attached-user-policies --user-name $user \
    --query "AttachedPolicies[?PolicyArn=='arn:aws:iam::aws:policy/AdministratorAccess']"
done

# Remove admin policy from user
aws iam detach-user-policy --user-name <user> \
  --policy-arn arn:aws:iam::aws:policy/AdministratorAccess

# Instead, create an admin role for JIT assumption
aws iam create-role --role-name AdminRole --assume-role-policy-document file://trust-policy.json
aws iam attach-role-policy --role-name AdminRole \
  --policy-arn arn:aws:iam::aws:policy/AdministratorAccess
```

### Check 1.4.1-B: Access Keys Older Than 90 Days

| Field | Value |
|-------|-------|
| Severity | HIGH |
| AWS API | `iam generate-credential-report`, `iam get-credential-report` |
| Fields Checked | `access_key_1_last_rotated`, `access_key_2_last_rotated` |

**What it checks:** Identifies active access keys that have not been rotated in over 90 days.

**Why it matters:** Long-lived credentials indicate:
- Keys are not being managed through a PAM/vault solution
- No automated rotation in place
- Higher risk if credentials are compromised

PAM solutions typically rotate credentials automatically or require check-out/check-in that triggers rotation.

**Pass condition:** All active access keys are less than 90 days old

**Fail condition:** One or more active access keys are older than 90 days

**Remediation:**
```bash
# List access keys for a user
aws iam list-access-keys --user-name <user>

# Create new access key
aws iam create-access-key --user-name <user>

# Deactivate old key (after updating applications)
aws iam update-access-key --user-name <user> --access-key-id <old-key> --status Inactive

# Delete old key
aws iam delete-access-key --user-name <user> --access-key-id <old-key>
```

### Check 1.4.1-C: Secrets Manager in Use

| Field | Value |
|-------|-------|
| Severity | LOW |
| AWS API | `secretsmanager list-secrets` |
| Fields Checked | Count of secrets |

**What it checks:** Whether AWS Secrets Manager is being used for credential storage.

**Why it matters:** Secrets Manager provides:
- Centralized credential storage (partial PAM capability)
- Automatic rotation support
- Audit logging via CloudTrail
- Fine-grained access control

While not a full PAM solution, Secrets Manager usage indicates the organization is centralizing credential management rather than hardcoding or manually managing secrets.

**Pass condition:** At least one secret exists in Secrets Manager

**Fail condition:** No secrets in Secrets Manager (may indicate credentials stored elsewhere or hardcoded)

**Note:** This is a LOW severity finding because:
- Organization may use external vault (HashiCorp Vault, CyberArk)
- Not all environments require Secrets Manager

**Remediation:**
```bash
# Create a secret
aws secretsmanager create-secret --name MyDatabasePassword \
  --secret-string '{"username":"admin","password":"secret123"}'

# Enable automatic rotation (requires Lambda function)
aws secretsmanager rotate-secret --secret-id MyDatabasePassword \
  --rotation-lambda-arn arn:aws:lambda:region:account:function:rotation-function \
  --rotation-rules AutomaticallyAfterDays=30
```

### Check 1.4.1-D: Admin Roles vs Admin Users Ratio

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `iam list-users`, `iam list-roles`, policy checks |
| Fields Checked | Count of admin users vs admin roles |

**What it checks:** Compares the number of IAM users with admin privileges to IAM roles with admin privileges.

**Why it matters:** IAM roles provide temporary credentials (typically 1-12 hours), which aligns with JIT access principles. IAM users have permanent credentials. A healthy PAM-aligned configuration should have:
- Few or zero admin IAM users
- Admin access via role assumption

**Pass condition:** More admin roles than admin users, OR zero admin users

**Fail condition:** More admin users than admin roles (indicates permanent admin access is the norm)

**Remediation:**
```bash
# Create admin role
aws iam create-role --role-name AdminRole \
  --assume-role-policy-document '{
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Allow",
      "Principal": {"AWS": "arn:aws:iam::ACCOUNT:root"},
      "Action": "sts:AssumeRole",
      "Condition": {"Bool": {"aws:MultiFactorAuthPresent": "true"}}
    }]
  }'

# Attach admin policy to role
aws iam attach-role-policy --role-name AdminRole \
  --policy-arn arn:aws:iam::aws:policy/AdministratorAccess

# Users assume role for admin tasks
aws sts assume-role --role-arn arn:aws:iam::ACCOUNT:role/AdminRole \
  --role-session-name AdminSession
```

### Info 1.4.1-E: Programmatic-Only IAM Users (Informational)

| Field | Value |
|-------|-------|
| Severity | INFO (not a finding) |
| AWS API | `iam get-credential-report`, `iam list-access-keys` |
| Fields Checked | `password_enabled=false` AND has active access keys |

**What it checks:** Identifies IAM users that have access keys but no console password (programmatic access only).

**Why it's informational:** These users *may* be:
- Service accounts (should migrate to IAM roles)
- CI/CD pipeline users (should migrate to IAM roles or OIDC federation)
- Developers who only use CLI (legitimate, but should have MFA on keys)
- Legacy accounts that need review

We cannot definitively determine which category each user falls into, so this is presented as information for manual review rather than a pass/fail finding.

**What to do with this information:**
1. Review each listed user
2. For service accounts: migrate to IAM roles with instance profiles or OIDC
3. For CI/CD: use OIDC federation (GitHub Actions, GitLab, etc.) or IAM roles
4. For human CLI users: ensure they have MFA and consider SSO with CLI access
5. Document any that must remain as IAM users with justification

**Remediation for service accounts:**
```bash
# Instead of IAM user with access keys, use IAM role
# For EC2:
aws iam create-instance-profile --instance-profile-name MyServiceProfile
aws iam add-role-to-instance-profile --instance-profile-name MyServiceProfile --role-name MyServiceRole

# For Lambda/ECS: use execution roles (automatic)

# For CI/CD (GitHub Actions example):
# Configure OIDC provider and role trust policy
```
