# Activity 2.7.1 - Implement EDR Tools and Integrate with C2C

## ZIG Requirement Summary

Per NSA ZIG Phase One, Activity 2.7.1 requires:
- EDR tooling implemented across all endpoints
- Critical EDR data sent to C2C (Comply-to-Connect) for device/user checks
- EPP (Endpoint Protection Platform) covers maximum services/applications
- Real-time monitoring for malware, ransomware, unauthorized access, behavioral anomalies
- Automatic quarantine, process termination, and alert generation

## Predecessor/Successor Relationships

```
2.3.4 (NextGen AV + C2C) → 2.7.1 (EDR + C2C) → 2.7.2 (XDR + C2C)
```

**Predecessor:** Activity 2.3.4 - NextGen AV integration with C2C must be in place first.

## AWS Limitations

AWS provides EDR-like capabilities through GuardDuty but cannot:
- Deploy agents to workstations/laptops/mobile devices
- Integrate directly with C2C (DoD-specific solution)
- Provide full EPP coverage (antivirus, DLP, encryption enforcement)
- Perform automatic endpoint quarantine or process termination

These checks verify AWS-native threat detection is enabled. Full EDR/EPP compliance requires dedicated endpoint security solutions.

## Checks Implemented

---

### Check 2.7.1-A: GuardDuty Runtime Monitoring

| Field | Value |
|-------|-------|
| Severity | MEDIUM/HIGH |
| AWS API | `guardduty list-detectors`, `guardduty get-detector` |
| Requirement | "Runtime threat detection" |

**What it checks:** Verifies GuardDuty Runtime Monitoring is enabled for EC2 and/or EKS.

**Why it matters:** Runtime Monitoring provides EDR-like capabilities by analyzing OS-level events, process execution, and network activity on EC2 instances and EKS containers.

**Pass condition:** RUNTIME_MONITORING or EKS_RUNTIME_MONITORING feature is ENABLED

**Fail condition (HIGH):** GuardDuty not enabled at all

**Fail condition (MEDIUM):** GuardDuty enabled but Runtime Monitoring disabled

**Remediation:** Enable GuardDuty Runtime Monitoring feature

---

### Check 2.7.1-B: GuardDuty Malware Protection

| Field | Value |
|-------|-------|
| Severity | MEDIUM |
| AWS API | `guardduty get-detector` |
| Requirement | "Malware detection" |

**What it checks:** Verifies GuardDuty EBS Malware Protection is enabled.

**Why it matters:** Malware Protection scans EBS volumes for malware when GuardDuty detects suspicious activity, providing automated malware detection without agents.

**Pass condition:** EBS_MALWARE_PROTECTION feature is ENABLED

**Fail condition:** Malware Protection not enabled

**Remediation:** Enable EBS Malware Protection in GuardDuty settings

---

### Check 2.7.1-C: GuardDuty Malware/Trojan Findings

| Field | Value |
|-------|-------|
| Severity | BLOCKER |
| AWS API | `guardduty list-findings` |
| Requirement | "Threat response" |

**What it checks:** Identifies active malware and trojan findings in GuardDuty.

**Why it matters:** Malware findings indicate active compromise and require immediate investigation and response.

**Pass condition:** No malware/trojan type findings

**Fail condition:** Active malware/trojan findings detected

**Finding types checked:**
- Trojan:EC2/DropPoint
- Trojan:EC2/BlackholeTraffic
- Trojan:EC2/DGADomainRequest.B/C
- Trojan:EC2/DNSDataExfiltration
- Trojan:EC2/PhishingDomainRequest

**Remediation:** Investigate immediately; isolate affected instances

---

### Check 2.7.1-D: CloudWatch Agent Configuration

| Field | Value |
|-------|-------|
| Severity | LOW |
| AWS API | `ssm get-parameter` |
| Requirement | "System-level monitoring" |

**What it checks:** Verifies CloudWatch Agent configuration exists in SSM Parameter Store.

**Why it matters:** CloudWatch Agent collects system-level metrics and logs that complement GuardDuty's threat detection with operational visibility.

**Pass condition:** AmazonCloudWatch-linux or AmazonCloudWatch-windows parameter exists

**Fail condition:** No CloudWatch Agent configuration in SSM

**Remediation:** Configure CloudWatch Agent and store config in SSM Parameter Store

---

## Checks NOT Implemented

| Requirement | Reason |
|-------------|--------|
| C2C integration | DoD-specific solution, not AWS-native |
| Full EPP coverage | Requires dedicated endpoint agent |
| Workstation/laptop EDR | Requires external EDR solution |
| Automatic endpoint quarantine | Requires EDR agent capabilities |
| SIEM/SOAR integration verification | Requires external platform access |
| EPP coverage assessment | Requires endpoint inventory correlation |

## External Tools for Full Compliance

To fully implement Activity 2.7.1, integrate AWS with:
- **EDR Solution** - CrowdStrike, Carbon Black, Microsoft Defender for Endpoint, SentinelOne
- **C2C** - Comply-to-Connect for device posture enforcement
- **SIEM** - Splunk, Elastic, etc. for centralized alerting
- **SOAR** - For automated incident response workflows

---

## How AWS Complements External EDR

While GuardDuty is not a full EDR solution, it provides cloud-layer threat detection that complements endpoint EDR:

| AWS Service | Complements |
|-------------|-------------|
| GuardDuty | Cloud-layer threat detection feeds SIEM |
| GuardDuty Runtime Monitoring | Container/EC2 behavioral analysis |
| GuardDuty Malware Protection | EBS volume scanning |
| Security Hub | Aggregates findings from EDR via integrations |
| CloudWatch Logs | Centralized log collection for correlation |

| EDR Capability | GuardDuty Feature |
|----------------|-------------------|
| Behavioral analysis | Runtime Monitoring |
| Malware detection | EBS Malware Protection |
| Network anomaly detection | VPC Flow Log analysis |
| Credential abuse detection | CloudTrail analysis |
| Container security | EKS Runtime Monitoring |

GuardDuty provides cloud infrastructure threat detection while external EDR handles endpoint-level protection. Together they provide defense in depth.
